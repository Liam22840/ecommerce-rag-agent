# 下单确认与收货地址 (Order Confirmation & Address)

本文档说明下单确认闭环里的收货地址能力，以及为支撑这个闭环而做的「上下文意图路由」设计。对应 spec 4.1 ⭐⭐⭐「下单确认流程：Agent 引导用户确认收货地址、汇总订单信息，模拟完成下单闭环」。

## 现状与那个曾经的死路

确认闭环本身是完整的：`_checkout` 汇总订单（件数 + 金额 + 地址），客户端 `OrderCardView` 渲染 待确认 / 已提交 / 已取消 三态，确认下单 / 取消按钮替用户发送「确认」/「取消订单」从而由服务端订单状态驱动，`_confirm_order` 校验草稿后生成模拟订单号并提交，取消与重新结算也都在。

唯一的缺口是**收货地址**：客户端有一个可编辑的「详细地址」，但它从不被发送给服务端，服务端 `OrderDraft.address` 永远是硬编码的「默认地址」。也就是说用户「确认」的地址对订单没有任何影响。本次把它端到端打通。

## 设计：地址是会话状态，单一真相

地址放在 `OrderState.address`（`server/commerce.py`）——它本来就被穿进每个 commerce 入口、并随会话持久化，所以不需要把地址再额外穿过一长串方法签名，和购物车状态的流转方式一致。

两个输入喂同一个值：

1. **客户端字段**：客户端把当前地址放进 `client_context.address` 每轮发送，所以订单从一开始就带着真实地址（默认是一个北京地址，而不是「默认地址」）。
2. **对话改地址**：一个 `set_address` 动作为这一轮覆盖它。因为 commerce 在每轮赋值之后才跑，所以「口头改地址」在本轮胜出；服务端随后把新地址回写到订单上，客户端再把自己的字段同步过去，两者永不分叉。

`_checkout` / `_confirm_order` 都读 `OrderState.address`，盖到 `OrderDraft` 和摘要文案上。

## set_address 动作（对话式改地址）

和购物车管理（「把数量改成 2」）同一个套路——LLM 理解语言、确定性执行：

- `set_address` 是白名单 `ActionType` 里的一个动作，由 `COMMERCE_INTENT_SYSTEM` 解析：用户要设置 / 修改收货地址（「把地址改成上海…」「寄到…」「送到…」）时，`action=set_address`，把地址原文照抄进 `address`（只照抄，不编造、不补全）。
- `_apply` 的 `set_address` 分支：写入 `order_state.address`；若存在待确认草稿，则重跑 `_checkout` 把卡片和摘要刷新成新地址（购物车没变，签名仍匹配，「确认」照样提交）；没有草稿时只记下地址，等下单时用。
- LLM 不可用时，确定性解析器**不**合成 `set_address`（地址无法用关键词抽取），这一路降级到客户端字段，符合「LLM 理解 → 确定性兜底」的原则。

## 客户端（Swift）

- `ChatViewModel` 持有 `shippingAddress`，随每个请求发送；收到订单事件时从 `order.address` 回同步，所以一次对话式「改地址」会让卡片字段跟着更新。
- `OrderCardView` 在 待确认 态展示一个可编辑地址行（复用既有的 `EditableOrderField`），「确认」沿用既有发送路径，把地址一起带上。
- 地址显示侧本来就已就绪：客户端 `Order` 模型已有 `address`，卡片渲染服务端 `summary`，所以一旦地址被发上去并被服务端采用，卡片会自动显示真实地址。

## 上下文意图路由（让「确认」不被误判为搜索）

打通确认流时发现一个路由问题：订单提交后，用户再随口说一句「确认」，会被路由到 `product_search`，于是搜索对「确认」这种没有真实诉求的输入返回最近邻商品（看上去就是「随机商品」）。这是一类问题的代表：**任何空洞 / 无诉求的输入只要落进搜索，就会被回一堆随机商品。**

我们刻意做了取舍，记录在此以免重蹈：

- **相关度阈值不可行（有数据为证）**：我们用约 100 条 query 标定了向量相似度。真实 query 的相似度区间和垃圾 / 不在售区间完全交错——真实的「冬天穿的外套」是 0.32，比垃圾的「好的」0.41 还低，比不在售的「冰箱」0.34 也低。任何能挡住「好的」的阈值都会同时误杀真实搜索。lexical 信号同样不可靠（「好的」也能命中 12 个商品的单字）。所以**没有任何相似度阈值能分开它们**——只有理解语义的 LLM 能分。
- **关键词枚举是打地鼠**：把「确认 / 好的 / 可以…」列进 prompt，能挡住列了的词，挡不住没列的（如「看看有什么」），换个说法就漏。

最终的做法是给路由器补上它缺的那点上下文，让它**推理**而不是匹配字符串：

- 路由器原本只拿到 raw query + 几个布尔（购物车有没有商品、有没有待确认订单、刚不刚对比过）。这是经典的 **dialogue state tracking**：把对话压缩成结构化状态喂给分类器。
- 新增 `last_route`（上一轮的意图类别，映射成可读标签如「处理下单或订单」「找/展示商品」）作为又一个压缩信号，和 `有待确认订单` 同样的轻量方式穿进路由器。
- 路由器提示改为：先判断这句话有没有真正的购物诉求；结合 `上一轮` 与 `有待确认订单` 推理——刚下完单后的附和 / 致谢 / 确认归 chitchat，想随意浏览（「随便看看」）仍归 search。

效果（真·LLM 验证）：空洞词归 chitchat，**包括从没列进 prompt 的词**（得嘞 / ok啦 / 嗯哼）——证明它泛化了，不是又一份词表。真实搜索照常，真实确认照常提交，而像「有什么推荐」这种边界问句会落到一句「您想看哪一类」的澄清，而不是回一堆随机商品。这也正是处理「不可消除的歧义」的成熟做法：**该问就问，而不是猜。**

## 测试 / 验证

- `tests/test_commerce_flow.py`：`set_address` 在有草稿时刷新待确认订单且摘要含新地址，确认后提交订单带上该地址；无草稿时只记录。
- 真·LLM 端到端：下单 → 摘要「收货地址为 X」；「把地址改成上海…」→ 卡片刷新成新地址并再次请确认；确认下单 → 提交订单带上它；客户端发送地址也能落到订单上。
- 路由：空洞词（含未列词）归 chitchat 不搜商品；真实 query 仍搜索；真实确认仍提交；边界浏览问句归澄清。
- iOS 侧需在 Xcode 构建后点测可编辑地址卡片。
