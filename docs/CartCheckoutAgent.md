# 购物车与下单 (Cart & Checkout Agent)

这份讲怎么把「把第一个加入购物车」「数量改成 2」「下单」「确认」这类大白话，变成购物车和订单上确定无误的操作。重点不在生成漂亮的回话，而在**稳、准、不出错**：金额、SKU、库存、订单号这些一律来自真实数据，模型说了不算。

## 它管哪些事

加购、删除、改数量、加一件 / 减一件、清空、查看购物车、设置收货地址、下单、确认、取消。下单是模拟的（会生成一个假订单号），但购物车金额、库存扣减、地址都是真的走完整流程。

## 怎么听懂购物车指令

和全局规矩一样：**模型先听懂，代码再核对和执行，模型不可用才用规则顶上。**

1. **模型主解析**：模型读用户原话 + 当前购物车 + 这一轮展示过的商品 + 有没有待确认订单，填出一个动作结构（哪件商品、什么动作、几件、什么规格、地址）。它只能填这个结构，**不许输出价格、SKU 价、订单金额**。
2. **规则兜底**：模型不可用时，退回关键词匹配（「加入购物车」→加购、「删掉」→删除、「下单」→结算……）。快、稳、对常见短句准。像收货地址这种没法靠关键词抽的，兜底时就不硬猜（这一轮交给前端的地址栏），符合全局规矩。
3. **代码核对并执行**：不管动作来自模型还是规则，最后都由代码把关——被指的商品必须是真展示过的、数量取整、加购数量按库存卡上限、金额和 SKU 全部来自定价模块。模型提议，后端算账。

> **技术细节**：`server/commerce.py` 的 `maybe_handle`（先 `_llm_parse`，不可用再 `_deterministic_parse`）→ `_apply`（校验 + 执行）；定价在 `server/pricing.py`。

## 「第一个」「那个」指的是谁

用户很少报商品全名，多是「第一个」「这个」「更便宜那个」。解析规则：

- **加购时**的「第一个 / 这个」，指最近展示过的那批商品（来源是前端带来的最近商品，或服务端自己的记忆，两边都有时以记忆为准，重启也不丢）。
- **删除 / 改数量时**的「第一个」，指购物车里的第几件。
- 购物车只有一件时，「再加一件」「数量改成 2」可以省略说哪一个。
- 购物车有多件但用户没说哪一个，就**反问**一句让他指明（这次没执行的动作会先记下来，用户回一句「1」「第一个」就接着把它做完）。

还有几个确定性的护栏，防止把模糊的话落到错的商品上：

- 用户像在指「对比胜出的那个」，但这一轮还没对比过 → 不去硬塞一个商品，而是反问。
- 「都加入 / 全部加入」只作用在**最近一批**展示的商品，不是这一整个会话见过的所有商品。
- 「换成家庭装」这种只换规格、没说数量的 → 数量不变，按新规格重新定价。

## 下单、确认、改地址

说「下单」就生成一个待确认的订单（汇总商品、金额、收货地址），说「确认」才真正提交（生成订单号、清空购物车），说「取消」就清掉草稿但留着购物车。

这里有个细节：确认时会比对购物车有没有变过。如果下单后用户又改了购物车再说「确认」，那就**重新出一张草稿**让他再确认一次，而不是把过期的旧单提交了。

**收货地址**走的也是「模型听懂、代码执行」那一套，和「把数量改成 2」同一个路子。地址作为会话状态存着，前端每一轮都把当前地址带上来，所以订单一开始就有真实地址（不是占位的「默认地址」）。用户中途说「把地址改成上海徐汇区…」，模型解析出来、代码写进会话、并刷新待确认的草稿；客户端再从订单回传的地址同步自己的输入框，两边始终一致。

> **技术细节**：会话态在 `OrderState`（草稿、购物车签名 `cart_signature`、挂起动作 `pending_action`、库存台账 `stock_sold`、地址 `address`）；`_checkout` / `_confirm_order` 都用 `order_state.address`；签名一致才提交、否则重出草稿。

## 库存

**先说清楚：库存是合成的。** 原始数据集没有库存字段，所以加载时按商品 id 确定性地造一个数（稳定落在 12–59 之间），再把两个好认的商品钉成低库存和售罄给演示用（iPhone = 2，兰蔻小黑瓶 = 0）。库存数字本身是演示值，但**建在它上面的工程是真的**：单一真相源 → 如实播报 → 加购拦截 → 下单扣减，这几层都成立。

- **台账**：会话里记着每个商品已下单多少，可用量 = 造出来的库存 − 已下单。待结算的购物车不算扣减。
- **如实播报**：交给模型的商品事实里带一个可用量字段，模型只能照抄，为 0 就说售罄、不当首选推荐，不许编库存数字。
- **加购拦截**：加购 / 加一件 / 改数量时，结果数量会被卡在可用量以内，售罄的直接跳过，超买的加到上限并说明。另外单行还有个数量上限（默认 999），挡住「要一百万件」这种。
- **实时扣减**：确认下单时把买的数量记进台账，之后同一会话再加这个商品、或重新搜到它，看到的就是减少（或售罄）后的量。扣减只在会话内，一次演示不会把全局掏空。

> **技术细节**：库存播种 `_seed_stock`（`12 + sha1(pid) % 48`），钉死值在 `_STOCK_PINS`；可用量 `_available` / `_available_by_id`；上限 `MAX_CART_QUANTITY`（`pricing.py`，可用环境变量覆盖）；扣减在 `_confirm_order`。

## 和多步任务的关系

单步的「把第一个加到购物车」直接走这个模块。复合的「推荐跑鞋并把最便宜的加入购物车」由多步任务模块（见 [`PlannerAgent.md`](PlannerAgent.md)）调度，但它不重写购物车逻辑，只是把确定好的真实商品 id 交给这个模块去执行，金额、SKU、库存照样统一走这里。

## 几个能复现的例子

- 展示三款面霜后说「把第一个加入购物车」→ 购物车出现第一款，数量 1，价格和卡片一致。
- 购物车多件时说「再加一件」→ 反问「想调整哪一件」，用户回「1」→ 第一件 +1。
- 想加兰蔻小黑瓶（售罄）→ 回「库存不足，暂时无法加入」，不入车。
- 「下单」→ 待确认；「把地址改成上海徐汇区…」→ 草稿刷新成新地址；「确认」→ 提交、出订单号、购物车清空。

> **技术细节**：黑箱测试在 `tests/test_commerce_flow.py`，覆盖指代解析、指定数量、增删改清、多件反问、挂起动作续接、结算确认取消、售罄拦截、误路由防护等。
